# Spec Critique: 13-robot-framework (Codex)

This critique focuses on scope compliance, standalone test observability, prerequisite failure handling, and lab lifecycle safety. I found no `CRITICAL` issues, but there are several `HIGH` gaps that should be resolved before implementation starts.

## HIGH

### 1. The draft does not line up with issue `#13`'s actual acceptance criteria or scope
- **Evidence:** Issue `#13` requires standalone Robot coverage for VLAN modes, DHCP lease acquisition for v4 and v6, PPPoE session establishment, validation errors including missing capabilities, SIGTERM cleanup, partial-failure cleanup, matrix execution across subscriber images, structured reports, a documented interop point for future osvbng Robot use, and local/CI usability. It explicitly marks osvbng-side integration and BNG-in-the-loop end-to-end tests out of scope. The spec instead introduces a full integration tier (`IMPLEMENTATION_SPEC.md:20-27`) and a BNG lab suite (`IMPLEMENTATION_SPEC.md:42-43`, `IMPLEMENTATION_SPEC.md:120-145`, `IMPLEMENTATION_SPEC.md:220-226`, `IMPLEMENTATION_SPEC.md:255-260`). The planned standalone suites cover env-var validation, VLAN creation, and cleanup only (`IMPLEMENTATION_SPEC.md:83-118`); there is no planned DHCPv6 lease test, PPPoE session test, missing-capability test, or concrete partial-failure rollback case. The matrix section also only shows `01-entrypoint-validation` commands per image (`IMPLEMENTATION_SPEC.md:230-236`, `IMPLEMENTATION_SPEC.md:248-252`).
- **Impact:** Implementing this spec would spend substantial effort on behavior the issue says not to build, while still leaving required acceptance items unmet. Phase 5 could "finish the spec" and still not close issue `#13`.
- **Recommendation:** Re-scope this issue around the required standalone Robot coverage. If you want BNG-in-the-loop Robot tests, split them into a follow-up issue that depends on `lab/`. In this spec, add explicit cases for DHCPv6 acquisition, PPPoE session establishment, missing `NET_ADMIN` / VLAN capability failures, and a real partial-failure cleanup scenario. Add a concrete documentation artifact for future osvbng interop instead of assuming the spec itself is sufficient.

### 2. `02-vlan-modes` does not define a harness that can reliably observe interface creation before DHCP exits
- **Evidence:** The only interface-existence check planned is `docker exec ip link show <iface>` (`IMPLEMENTATION_SPEC.md:72`). The suite note says the test can verify interfaces either with "a sidecar check" or "by inspecting the container state after a short delay" (`IMPLEMENTATION_SPEC.md:108`), but the spec never defines either mechanism. The entrypoint creates VLANs and then immediately launches DHCP and waits for the client process (`images/shared/entrypoint.sh:157-188`, `images/shared/entrypoint.sh:213-234`).
- **Impact:** `docker inspect` cannot tell you whether `eth0.100` or `eth0.100.10` existed inside the container network namespace, and a vague "sidecar" is not enough to implement a deterministic Robot keyword. The suite will race the DHCP timeout and produce flaky results instead of proving interface creation.
- **Recommendation:** Spell out a concrete harness: run the container detached, set `DHCP_TIMEOUT` high enough for inspection, poll `docker exec ip link show` while the container is still running, and fail immediately if the container exits before the target interface appears. Remove the "inspect container state after a short delay" option unless you also define a persistent external namespace-based observer.

### 3. `03-cleanup` cannot prove VLAN removal with the planned `Check Interface Removed` keyword
- **Evidence:** The cleanup suite expects to confirm that VLAN interfaces were removed after SIGTERM or DHCP timeout (`IMPLEMENTATION_SPEC.md:114-118`). The only removal check in the keyword plan is `docker exec ip link show <iface>` returning non-zero (`IMPLEMENTATION_SPEC.md:79`). But cleanup is driven by the entrypoint's `EXIT` trap (`images/shared/entrypoint.sh:35-68`). Once the container exits, its network namespace disappears and `docker exec` can only report that the container is no longer running, not whether cleanup successfully deleted the VLANs before teardown.
- **Impact:** The suite can pass for the wrong reason. A clean shutdown, an early crash, and a hard kill all look identical after the namespace is gone.
- **Recommendation:** Redesign the cleanup tests around a persistent namespace owner or another harness that keeps the network namespace alive after the entrypoint process exits. If that is too heavy for this issue, scale the assertion back to observable evidence such as log ordering and explicit exit codes, and document that it is weaker than true post-cleanup interface inspection.

### 4. `04-ipoe-bng` has no safe lifecycle for an existing or partially failed lab
- **Evidence:** `Deploy Topology` is defined as `sudo containerlab deploy -t <file> --reconfigure` and `Destroy Topology` as `sudo containerlab destroy --cleanup` (`IMPLEMENTATION_SPEC.md:55-56`), and suite `04-ipoe-bng` says setup deploys the lab and teardown destroys it (`IMPLEMENTATION_SPEC.md:135`). The topology name is fixed to `bngtester` (`lab/bngtester.clab.yml:5`), which means the container names are always `clab-bngtester-*` (`lab/smoke-test.sh:12-16`). The current lab workflow keeps deploy, validate, and destroy as separate user-controlled steps (`lab/README.md:22-40`; `lab/smoke-test.sh:6-8`). This also breaks the advertised image-matrix path: the spec says `${SUBSCRIBER_IMAGE}` maps to `BNGTESTER_IMAGE` for integration tests (`IMPLEMENTATION_SPEC.md:145`), but the current lab docs require `sudo -E` to preserve those env vars when deploying (`lab/README.md:47-58`).
- **Impact:** If a developer already has the lab up for debugging, this suite will reconfigure it and then tear it down on teardown. If a previous deploy partially failed, the spec does not say whether the suite should attach to the existing state, destroy it first, or abort with diagnostics. The integration image-matrix story is also incomplete as written because plain `sudo containerlab` will typically drop the override variables.
- **Recommendation:** If this suite remains, define a lab ownership contract: unique per-run lab name or attach mode, preflight detection of existing `clab-bngtester-*` state, recovery rules for partial deployments, and teardown that only destroys resources created by the current run. If image overrides remain part of the design, make the `sudo -E` requirement explicit or use another mechanism that preserves the variables.

## MEDIUM

### 5. The runner contract is missing basic prerequisite failure handling
- **Evidence:** `tests/rf-run.sh` is specified only as venv setup plus `robot` invocation (`IMPLEMENTATION_SPEC.md:149-155`). The dependency list names Docker, containerlab, and the osvbng image (`IMPLEMENTATION_SPEC.md:168-174`), but there is no preflight for a stopped Docker daemon, an unavailable `${SUBSCRIBER_IMAGE}`, missing `containerlab`, or the host prerequisites already documented for the lab such as image pulls and hugepages (`lab/README.md:17-20`, `lab/README.md:101-107`).
- **Impact:** Local and CI failures will show up as opaque `docker run` / `docker exec` / `containerlab` errors instead of actionable test failures or skips. That conflicts with the issue requirement that the suite be runnable both locally and in CI.
- **Recommendation:** Add explicit preflight keywords or runner checks: `docker info`, `docker image inspect` or controlled `docker pull` for `${SUBSCRIBER_IMAGE}`, `command -v containerlab`, `sudo -n true`, and integration-only host checks with clear fail/skip messages.
